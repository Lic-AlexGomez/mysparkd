[CRITICO-1] CREDENCIALES HARDCODEADAS EN CÓDIGO FUENTE
Archivo: SecurityConfig.java (líneas 44-47)
-----------------------------------------------
PROBLEMA:
  El usuario y contraseña del servidor de correo están escritos directamente
  en el código fuente:
    mailSender.setUsername("cotcejonesan1@gmail.com");
    mailSender.setPassword("lmsb btuq phsh xthw");

  Esto expone las credenciales a cualquiera que tenga acceso al repositorio.

SOLUCIÓN:
  Mover al archivo application.properties o variables de entorno:

  # application.properties
  spring.mail.username=${MAIL_USERNAME}
  spring.mail.password=${MAIL_PASSWORD}

  # SecurityConfig.java
  @Value("${spring.mail.username}") private String mailUsername;
  @Value("${spring.mail.password}") private String mailPassword;
  mailSender.setUsername(mailUsername);
  mailSender.setPassword(mailPassword);

  Y en el .env del servidor:
  MAIL_USERNAME=cotcejonesan1@gmail.com
  MAIL_PASSWORD=lmsb btuq phsh xthw

[CRITICO-2] JWT SIN FECHA DE EXPIRACIÓN
Archivo: JwtUtils.java (línea 54)
-----------------------------------------------
PROBLEMA:
  El token principal de autenticación no tiene fecha de expiración:
    //.setExpiration(new Date(System.currentTimeMillis() + 1000 * 60 * 60))

  Un token robado es válido para siempre. No hay forma de invalidarlo.

SOLUCIÓN:
  Descomentar y configurar la expiración. Recomendado: 7 días para móvil.

    .setExpiration(new Date(System.currentTimeMillis() + 1000L * 60 * 60 * 24 * 7))

  Si se necesita sesión más larga, implementar refresh tokens.

[CRITICO-3] CLAVE SECRETA JWT HARDCODEADA
Archivo: SecurityConfig.java (línea 65)
-----------------------------------------------
PROBLEMA:
  La clave secreta del JWT está escrita en el código:
    Keys.hmacShaKeyFor("clave-secreta-muy-larga-y-segura-para-jwt"
                       .getBytes(StandardCharsets.UTF_8))

  Esta clave es débil y está expuesta en el repositorio.

SOLUCIÓN:
  # application.properties
  jwt.secret=${JWT_SECRET}

  # SecurityConfig.java
  @Value("${jwt.secret}") private String jwtSecret;

  @Bean
  public SecretKey jwtSecretKey() {
      return Keys.hmacShaKeyFor(jwtSecret.getBytes(StandardCharsets.UTF_8));
  }

  Y en .env usar una clave aleatoria de al menos 64 caracteres:
  JWT_SECRET=<generar con: openssl rand -base64 64>

[CRITICO-4] CORS SOLO PERMITE LOCALHOST — PRODUCCIÓN BLOQUEADA
Archivo: CorsConfig.java (líneas 16-22)
-----------------------------------------------
PROBLEMA:
  Solo se permiten orígenes de localhost. El frontend en producción
  (mysparkd.com) está bloqueado por CORS:

                    "http://localhost:3001",
                    "http://localhost:8081")

SOLUCIÓN:

        "https://mysparkd.com"
    )
    .allowedHeaders("*")

[ALTO-1] PRESENCIA: STATUS "online" EN MINÚSCULA vs "ONLINE" EN MAYÚSCULA
Archivo: ChatService.java (línea ~200)
-----------------------------------------------
PROBLEMA:
  Cuando se envía un mensaje, se emite un evento de presencia con status
  en minúscula:
    Map.of("userId", sender.getUserId().toString(), "status", "online")

  Pero WebSocketEventListener emite "ONLINE" (mayúscula) y el frontend
  compara con .toUpperCase(). Esto causa inconsistencias en el indicador
  de "En línea".

SOLUCIÓN:
  Cambiar a mayúscula para ser consistente:
    Map.of("userId", sender.getUserId().toString(), "status", "ONLINE")

  O mejor aún, usar el PresenceDTO que ya existe:
    messagingTemplate.convertAndSend("/topic/presence",
        new PresenceDTO(sender.getUserId(), "ONLINE"));

[ALTO-2] GOOGLE LOGIN — USUARIO SIN CONTRASEÑA NI VALIDACIÓN
Archivo: AuthService.java (líneas ~110-145)
-----------------------------------------------
PROBLEMA:
  Al crear un usuario con Google OAuth, no se asigna contraseña ni se
  valida que el email sea válido. El username se setea como el email completo
  (con @), lo que puede causar conflictos:
    newUser.setUsername(payload.email()); // "usuario@gmail.com" como username

  Además, no hay @Transactional en el método, por lo que si falla la subida
  a Cloudinary, el usuario queda guardado sin foto pero sin rollback.

SOLUCIÓN:
  1. Generar username limpio desde el email:
     String username = payload.email().split("@")[0] + "_" + UUID.randomUUID().toString().substring(0,4);
     newUser.setUsername(username);

  2. Agregar @Transactional al método loginWithGoogle

  3. Manejar el error de Cloudinary sin romper el registro:
     try {
         // subir foto
     } catch (Exception e) {
         // continuar sin foto, no fallar el registro
     }

[ALTO-4] POLL SYNC — CARGA TODOS LOS POLLS CADA 30 SEGUNDOS
Archivo: PollService.java (línea ~185)
-----------------------------------------------
PROBLEMA:
  El job programado carga TODOS los polls de la base de datos cada 30 segundos:
    List<Poll> polls = pollRepository.findAll();

  Con muchos polls esto puede saturar la base de datos y causar timeouts.

SOLUCIÓN:
  Solo sincronizar polls activos (no expirados):
    List<Poll> polls = pollRepository.findByExpiresAtAfter(LocalDateTime.now());

  O aumentar el intervalo a 5 minutos:
    @Scheduled(fixedRate = 300000)

[ALTO-5] DELETE POLL — SOLO LIMPIA REDIS DEL USUARIO QUE BORRA
Archivo: PollService.java (línea ~170)
-----------------------------------------------
PROBLEMA:
  Al eliminar un poll, solo se borra la clave Redis del usuario que hace
  el delete, no de todos los usuarios que votaron:
    redisTemplate.delete("poll:" + poll.getId() + ":user:" + userId);

  Las claves de otros votantes quedan huérfanas en Redis.

SOLUCIÓN:
  Usar un patrón para borrar todas las claves del poll:
    // Borrar votos de opciones
    for (PollOption option : poll.getOptions()) {
        redisTemplate.delete("poll:" + poll.getId() + ":option:" + option.getId());
    }
    // Borrar todos los votos de usuarios (requiere scan o guardar lista de votantes)
    Set<String> keys = redisTemplate.keys("poll:" + poll.getId() + ":user:*");
    if (keys != null) redisTemplate.delete(keys);

[MEDIO-4] USO DE @AUTOWIRED EN VEZ DE INYECCIÓN POR CONSTRUCTOR
Archivo: AuthService.java (múltiples campos)
-----------------------------------------------
PROBLEMA:
  Se usa @Autowired en campos, lo que dificulta el testing y puede causar
  problemas con dependencias circulares:
    @Autowired private UsersRepository usersRepository;
    @Autowired private PasswordEncoder passwordEncoder;
    // etc.

SOLUCIÓN:
  Usar inyección por constructor (como ya se hace en ChatService, PostService):
    private final UsersRepository usersRepository;
    private final PasswordEncoder passwordEncoder;

                       PasswordEncoder passwordEncoder, ...) {
        this.usersRepository = usersRepository;
        this.passwordEncoder = passwordEncoder;
    }

[BAJO-1] PRESENCECONTROLLER — SIN LASTSEEEN EN RESPUESTA
Archivo: PresenceController.java
-----------------------------------------------
PROBLEMA:
  El endpoint GET /api/presence/{userId} no devuelve el campo lastSeen
  cuando el usuario está OFFLINE, aunque el dato existe en la base de datos.

SOLUCIÓN:
  Agregar lastSeen a la respuesta:

  @GetMapping("/{userId}")
  public ResponseEntity<Map<String, Object>> getPresence(@PathVariable UUID userId) {
      boolean online = presenceService.isOnline(userId);
      Map<String, Object> response = new HashMap<>();
      response.put("userId", userId.toString());
      response.put("status", online ? "ONLINE" : "OFFLINE");
      if (!online) {
          usersRepository.findById(userId).ifPresent(user -> {
              response.put("lastSeen", user.getLastSeen());
          });
      }
      return ResponseEntity.ok(response);
  }

[BAJO-4] RESOLVERESOURCETYPE — LÓGICA DUPLICADA
Archivo: PostService.java (método resolveResourceType)
-----------------------------------------------
PROBLEMA:
  El método tiene una verificación redundante — verifica format dos veces:
    if (post.getMedia().getFormat() != null) {
        return post.getMedia().getFormat(); // retorna el format crudo, no "image"/"video"
    }
    String format = post.getMedia().getFormat(); // nunca llega aquí

SOLUCIÓN:
  Simplificar el método:
    private String resolveResourceType(Posts post) {
        if (post.getMedia() == null) return "image";
        String url = post.getMedia().getMediaUrl();
        if (url != null && url.contains("/video/")) return "video";
        String format = post.getMedia().getFormat();
        if (format != null) {
            format = format.toLowerCase();
            if (List.of("mp4","mov","avi","webm").contains(format)) return "video";
        }
        return "image";
    }
